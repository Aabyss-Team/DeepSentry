## 当前版本重点变化

| 能力 | 用户该怎么理解 |
|------|----------------|
| 模型与上下文向导 | `--init` 内置千帆 Coding Plan、火山方舟 Coding Plan、MiMo Token Plan/MiMo Claw 和自定义接口；可选 auto、64K、128K、256K、512K、1M、2M 或自定义窗口 |
| 工具参数契约 | 内置工具在执行前校验 action、必填参数、枚举和类型；不确定时先调用 `tool_catalog` 获取可执行示例 |
| SSH 主机身份 | 默认 `accept-new` 在首次连接时锁定密钥，后续密钥变更会被拒绝；正式环境不要使用 `insecure` |
| 安全归档 | 本地解压/打包防路径穿越、链接越界和解压炸弹；远程直接解压失败关闭，应先下载到控制端处理 |
| 初始化三种模式 | 首次 `--init` 可选择本地、远程 1 台、远程多台 Fleet |
| Shell-first 执行 | 默认优先用目标机原生 Shell；内置工具用于 Shell 不够稳、需要结构化解析或控制端探测的场景 |
| 多目标 Subagent | 对 `targets[]` 批量任务时，每台服务器使用独立连接上下文，避免串目标 |
| 分层长上下文 | 固定保留原始目标、最新用户修正、上一版摘要、核心线索与最近步骤；摘要失败也能继续 |
| 核心线索板 | 自动汇聚 IP、URL、CVE、哈希、路径和明确结论，可用 `/memory clues` 查看 |
| 并发协作 | 子 Agent 获得主目标、TODO 和线索板；任务去重、受控并发，并向主 Agent 结构化交接 |
| TUI 滚动与输入 | 向上翻阅历史时隐藏输入区并扩大日志视口；回到底部后恢复输入，同时保持中文输入法光标锚点稳定 |
| 长文本粘贴 | 可粘贴大段代码/日志，界面折叠显示，发送给 AI 的仍是完整内容 |
| TUI 斜杠命令 | 支持会话、Memory、Skills、MCP、TSecBench、配置和状态管理 |
| no-tui 思考流 | 默认显示可读思考摘要，不再把完整 JSON action 当思考内容刷屏 |
| 询问与格式恢复 | 模型需要补充信息或要求选择时进入询问面板；普通 Markdown、Shell 代码块和残缺结构按类型恢复或重试 |
| Emoji/表格对齐 | 按完整字素簇计算宽度，修复组合 Emoji 与文字重叠、Markdown 表格和询问框右边线错位 |
| sudo 安全交互 | 本机交给系统 `sudo -v` 隐藏验证，远程固定非交互执行；密码不进入输入框、日志或模型上下文 |
| 65 个内置工具 | 覆盖系统、网络、日志、文件、Web、数据库、取证、Fleet、配置、调度和代理转发 |

## 1. 快速开始

### 1.1 安装与编译

```bash
git clone https://github.com/asaotomo/DeepSentry.git
cd DeepSentry
go build -o deepsentry ./cmd/main.go ./cmd/usage.go ./cmd/survey_compat.go ./cmd/console_other.go
# Windows: 使用 console_windows.go 替代 console_other.go
```

### 1.2 首次运行（配置向导）

```bash
./deepsentry --init
```

向导会询问模型/API 的实际上下文长度，可选自动、64K、128K、256K、512K、1M、2M 或自定义。自定义范围为 4096-4194304 tokens。请以 API 实际限制或本地服务的 `num_ctx` / `max_model_len` 为准；不确定时选“自动”。

向导会引导你配置：

| 步骤 | 说明 |
|------|------|
| AI 服务商 | DeepSeek / Qwen / 千帆 Coding Plan / 火山方舟 Coding Plan / 电信星辰 / 腾讯混元 / OpenAI / Claude / Gemini / MiMo Token Plan / Ollama 等 |
| API 地址 | 支持只填 `/v1`，自动补全 `chat/completions` |
| 模型名称 | 如 `mimo-v2.5-pro`、`deepseek-v4-pro`、`qwen-plus`、`hunyuan-turbos-latest` |
| API Key | 本地 Ollama 可留空 |
| 最大轮数 | 默认 30，防止 Agent 死循环 |
| 管理模式 | 默认本地；Windows cmd 方向键无效时可用 Tab / Shift+Tab 切换 |
| 连接信息 | 根据模式录入 SSH/Telnet/FTP 或多台 `targets[]` |

配置保存至 `./config.yaml` 或 `~/.deepsentry/config.yaml`。

**管理模式说明：**

| 模式 | 结果 |
|------|------|
| 本地任务 | 清空远程连接字段，所有工具在控制端本机执行 |
| 远程管理 1 台服务器 | 写入旧版兼容字段：`target_protocol`、`ssh_*`、`telnet_*`、`ftp_*` |
| 远程管理多台服务器 | 写入 `targets[]`，主进程保持本地控制端模式，由 Fleet/subagent 逐台连接 |

### 1.3 推荐：TUI 模式

```bash
./deepsentry --tui
./deepsentry --tui "排查目标机内存与监听端口"
./deepsentry --plan "配置每 10 分钟 CPU 监控并通知"
./deepsentry --tui -c /path/to/config.yaml --resume session_xxx
```

**TUI 界面说明：**

| 按键 | 功能 |

**长文本粘贴：** 直接粘贴 1000 行代码/日志时，输入框显示紧凑的“粘贴文本”块，提交给 AI 的仍是完整内容。后续输入保持可见可编辑；多行/换行输入中 `↑` / `↓` 优先移动光标，到达边界后才切换历史。
**历史翻阅与输入框：** 向上滚动离开实时底部后，输入区会隐藏，日志视口会扩展；按 `Ctrl+End` / `G` 或滚回底部后恢复输入。提交新消息也会自动返回实时底部。
**中文输入法：** 位于实时底部时，真实光标锚点会跟随输入行；翻阅历史时隐藏真实输入，避免候选框或光标落到日志区域。若显示行为与此不符，请确认启动的是当前构建的同一个二进制。
**Markdown 与 Emoji：** 用户询问和最终报告使用同一套 Markdown 渲染；窄窗口中的表格会自动改为键值列表。Emoji 变体选择符、肤色和 ZWJ 组合不会被拆开计算，面板绘制前还会执行最终宽度裁剪。
**用户询问与模型格式兼容：** 模型提出问题、要求补充信息或提供选项时，TUI 会进入询问面板等待用户输入；不会把需要用户选择的内容当成最终报告。模型偶尔返回普通 Markdown 或 Shell 代码块时会按类型恢复，无法可靠恢复的残缺结构会触发重试。
**sudo 权限：** 本机 TUI 会在需要时暂停全屏，把终端交给系统 `sudo -v` 隐藏读取密码，验证完成后使用 `sudo -n` 执行原命令。验证成功、失败或取消后，程序会显式恢复备用屏幕、鼠标跟踪和全屏重绘，滚轮会继续翻阅 TUI 会话。密码不会进入输入框、日志、报告、Memory 或模型上下文。可在空闲时用 `/sudo` 主动刷新授权。Batch/WebShell 等非交互模式不会索要密码；远程目标只使用免交互 sudo，不会自动复用 SSH 密码。

### 1.4 经典 CLI 模式

```bash
./deepsentry --no-tui "分析 auth.log 中的暴力破解 IP"
./deepsentry --no-tui -c config.yaml --batch -y "自动巡检"
./deepsentry --no-tui --json --task "mem_info + port_listen 摘要"
./deepsentry --webshell --task "读取最近登录日志"
```

说明：

- 默认运行会进入 TUI；脚本、WebShell、CI 建议显式加 `--no-tui`。
- `--no-tui` 普通输出会显示 AI 思考摘要，不会把完整 JSON action 作为思考内容刷屏。
- `--json` 输出 JSONL 事件，适合外部程序消费。
- `--webshell` 会提交后台任务并立即返回报告/进度路径：后台子进程启用 `--no-tui --batch -y`，命令 stdout/stderr 会像 fscan 一样逐行写入 `reports/webshell_progress_<timestamp>.log`，完整结果仍写入报告并保留给 Agent；固定索引在 `reports/latest_webshell.txt`。
- WebShell 补充信息恢复示例：`./deepsentry --webshell --resume session_xxx --task "Webhook 是 https://..."`

交互输入时会显示**示例建议**（Tab/方向键可选取）：

- 排查目标机内存与监听端口
- 分析 SSH 登录失败记录
- 审计网络暴露面与异常连接
- 在 Web 目录狩猎 webshell

## 3. 工作模式

### 3.1 本地模式、远程单机、Fleet 多机

| 模式 | 触发条件 | 工具视角 |
|------|----------|----------|
| **本地** | 单机连接字段为空，且没有指定远程目标任务 | 所有操作在控制端本机 |
| **远程单机** | 配置 `target_protocol` + 对应 `ssh_*` / `telnet_*` / `ftp_*` | 🎯 目标机 / 💻 控制端 双视角 |
| **Fleet 多机** | 配置 `targets[]` | 主进程本地控制端，按 selector 为每台目标创建连接 |

- **目标机工具**（读 `/proc`、SFTP）：`mem_info`、`port_listen`、`process_list`、`read_file`…
- **控制端工具**（从 DeepSentry 进程发起）：`ping`、`dns_lookup`、`http_probe`、`nmap_scan`…

### 3.2 Fleet 多目标配置

```yaml
targets:
  - name: web-01
    protocol: ssh
    host: 10.0.0.11:22
    user: root
    password: ""
    key_path: ~/.ssh/id_ed25519
    tags: ["prod", "web"]

  - name: legacy-router
    protocol: telnet
    host: 10.0.0.2:23
    user: admin
    password: admin
    telnet_prompt: ">"
    tags: ["legacy", "network"]

  - name: ftp-backup
    protocol: ftp
    host: 10.0.0.50:21
    user: backup
    password: YOUR_PASSWORD
    tags: ["backup"]
```

selector 支持：

| selector | 命中规则 |
|----------|----------|
| `all` | 全部目标 |
| `web-01` | 按 name |
| `10.0.0.11` | 按 host 包含 |
| `ssh` / `telnet` / `ftp` | 按协议 |
| `prod` / `legacy` | 按 tag |

### 3.3 Fleet 与 target-aware 子 Agent

| 方式 | 适合场景 | 示例 |
|------|----------|------|
| `fleet_inventory` | 查看目标清单 | “列出所有 prod 目标” |
| `fleet_exec` | SSH/Telnet 批量执行只读巡检命令 | “对 prod 执行 uptime 和 df -h” |
| `fleet_file` | SSH/Telnet/FTP 文件操作 | “读取所有目标 /var/log/auth.log 末尾 100 行” |
| `task + target_selector` | 每台机器独立分析、最后汇总 | “让 log-analyst 分析所有 prod 机器登录日志” |

`task + target_selector` 示例：

```json
{
  "action": "task",
  "task_name": "log-analyst",
  "target_selector": "prod",
  "task_prompt": "审计最近 24 小时 SSH 登录日志，列出高风险 IP 和成功登录用户"
}
```

注意：

- 每个目标会创建独立 executor，不共享全局连接，避免多机并发串目标。
- 低风险只读任务可自动并发；写入、批量修改、高风险命令需要确认。
- 已配置 targets 时不要让 Agent 裸跑 `ssh/scp/sftp root@host`；这些命令不会读取 DeepSentry 配置里的密码/私钥，请使用 `fleet_exec`、`fleet_file` 或 `target_selector`。
- FTP 目标不能执行 shell，因此 `fleet_exec` 会拒绝 FTP；请使用 `fleet_file` 做 `ls/read/download/upload`。

### 3.4 Agent 动作协议

Agent 每步输出 JSON action，支持：

| action | 用途 |
|--------|------|
| `tool` | 调用内置 Go 原生工具 |
| `execute` | Shell 命令（高风险需确认） |
| `task` | 委派子 Agent；可带 `target_selector` 分解多目标任务 |
| `load_skill` | 加载安全 Skill |
| `todo` | 任务规划 |
| `ask_user` | 缺少关键信息或需要用户选择时，进入询问面板并暂停等待输入 |
| `read_file` / `grep` / `ls` / `glob` / `edit_file` | 文件操作 |
| `remember` / `forget` | 跨会话记忆 |
| `finish` | 完成任务 |

### 3.5 跨会话 Memory

DeepSentry 默认内置一份 `builtin://AGENTS.md`，随单二进制一起编译，不需要额外携带文件。

可选扩展：

- `~/.deepsentry/AGENTS.md`：用户级长期偏好
- `.deepsentry/AGENTS.md`：项目级规范
- `~/.deepsentry/memory/store.json`：`remember` / `forget` 写入的结构化记忆

Memory 会自动注入 Agent 上下文；明显的 API Key、密码、Token、私钥等敏感凭证会被拒绝保存。

长期记忆按固定上下文预算注入：优先保留内置安全规则、当前目标记忆和较新的条目，超出预算的内容会明确标注省略，不会无限挤占当前任务的推理空间。

会话内另有“核心线索板”，自动提取高信号的 IP、URL、CVE、哈希、文件路径和明确结论，并保留来源供 Agent 判断是用户提供、执行证据还是推断。它随 checkpoint 保存并提供给主 Agent/子 Agent 协作，但不会自动升级为跨会话永久记忆，避免把一次性噪声长期污染 Memory。

## 4. 安全应急场景指南

### 4.1 内存 / CPU 异常

```
排查目标机内存使用，列出前 20 个进程，给出结论
```

推荐工具链：`mem_info` → `process_list` → `finish`

### 4.2 网络暴露面 / 异常连接


使用：`{"action":"load_skill","skill_name":"forensics"}`

#### 外部 Skill 与启停管理

外部 Skill 默认从 `./skills` 和 `~/.deepsentry/skills` 加载。每个 Skill 使用独立目录，且目录内必须包含带 YAML frontmatter 的 `SKILL.md`。

常用命令：

```text
/skill list
/skill find log forensics
/skill inspect clawhub:security-audit
/skill install skills:owner/repo@skill-name
/skill updates
/skill update skill-name
/skill pin skill-name
/skill rollback skill-name
/skill load log-audit
/skill unload log-audit
/skill on log-audit
/skill off log-audit
/skill add /opt/deepsentry-skills
/skill source-off /opt/old-skills
/skill source-on /opt/old-skills
```

如果当前发现了多个 Skill，只想启用 `fofamap`，直接输入：

```text
/skill only fofamap
```

该命令会自动打开全局 Skill 开关、启用 `fofamap`、禁用其余当前已发现的 Skill，并立即应用到当前会话和持久化配置；不需要先执行 `/skill on`。执行 `/skill list` 可以核对最终状态。

`load` 只加载到当前会话；`on` / `off` / `only` 会修改持久化启停策略。市场安装前会进行来源和静态安全检查；搜索不会自动安装，存在风险提示时需要人工复核并显式确认。

| 名称 | 职责 |
|------|------|
| `log-analyst` | 日志取证 |

| `webshell-hunter` | Webshell 检测 |
| `network-analyst` | 网络流量分析 |
| `general-purpose` | 通用多步任务 |

| `awd-plus-operator` | AWD-plus 多服务运营 |

使用：`{"action":"task","task_name":"network-analyst","task_prompt":"..."}`

并行委派只适合相互独立的方向。运行器会自动去重相同任务、限制总体并发；每个子 Agent 收到主目标、TODO 和核心线索，并在后续步骤读取共享线索板的新证据。子 Agent 返回“任务状态、核心结论、关键证据、冲突/不确定项、建议下一步”，主 Agent 汇总后更新 TODO。有前后依赖的工作应分两批委派，不应强行放进同一个 `parallel_tasks`。

### 5.3 长上下文、核心线索与并发协作

#### 长上下文如何工作

DeepSentry 会自动管理历史，不需要用户手动复制前情：

| 层级 | 保留内容 |
|------|----------|
| 固定层 | 第一条真实用户目标、最新用户补充/修正、当前核心线索 |
| 摘要层 | 早期命令、关键输出、文件变更、失败原因、TODO、冲突和下一步 |
| 近期层 | `compact` / `balanced` / `full` 默认保留 8 / 12 / 24 条原文，巨大消息仍服从 token 预算 |
| 长期层 | `AGENTS.md` 和 `remember` 保存的跨会话规则/事实，按模型 profile 预算注入 |

当前版本按 `context_window_tokens × context_utilization - system prompt - 本轮工具 schema - 输出预留` 计算历史预算，不再使用固定 60,000 字节阈值。1M 窗口可继续积累大量有效历史；小窗口则更早压缩。摘要会覆盖所有早期分块，大段工具输出不再只保留首尾。

如果摘要模型失败、超时或被停止：

- 保留第一条用户目标和最新修正；
- 保留上一版成功摘要；
- 保留核心线索板；
- 在当前 token 预算内保留尽可能多的最近消息；
- 不会把失败的半成品摘要写回 checkpoint。

本地模型建议在 `config.yaml` 显式填写服务端实际窗口：

```yaml
model_profile: auto
model_parameter_b: 14          # 模型名已含 14b 时可留 0
context_window_tokens: 32768   # 对齐 Ollama num_ctx / vLLM max_model_len
context_utilization: 0         # 0=按 profile 自动
reserved_output_tokens: 0
native_tool_limit: 0
```

启动信息会显示实际 profile、context、输出预留和 Native Tool 直接数量。未声明窗口的本地服务会先按 32K 安全默认运行。

#### 核心线索板

核心线索板属于当前会话，最多保留 48 条高信号候选事实：

- IP、URL、CVE、哈希、Flag、文件路径；
- 明确标注为关键结论、证据、失败原因、冲突或高风险的短结论；
- 每条线索保留来源；同一线索来自多台目标或多个子 Agent 时会合并来源，不会被最后一次结果覆盖；
- 凭证、密码、Bearer Token、私钥等敏感内容不会进入线索板；
- 高优先级标识符会优先保留，普通结论不会无限增长。

查看或清空：

```text
/memory clues
/memory clues clear
```

`clear` 只清空当前会话线索，不会删除 `~/.deepsentry/memory/store.json`、`AGENTS.md` 或历史报告。核心线索会随 checkpoint 保存和恢复，但不会自动变成永久 Memory；需要跨会话保留时再使用 `remember`。

线索板内容是“带来源的候选事实”，不等于全部已经验证。主 Agent 会结合来源区分用户提供、工具证据和推断；跨目标线索未经当前目标复核，不应直接当作本机事实。

#### 多子 Agent 并发规则

当前调度边界：

| 类型 | 最大并发 | 说明 |
|------|---------:|------|
| 本地、彼此独立的子任务 | 4 | 日志、网络、文件等不同方向可并行 |
| 带 `target_selector` 的并行任务 | 3 | 控制远程连接和模型请求压力 |
| 并行任务内部的多目标展开 | 1 | 防止形成“外层并发 × 内层并发”的连接风暴 |

运行器还会：

- 忽略空任务，并对名称、任务内容、目标均相同的委派去重；
- 为每个子 Agent 使用独立执行历史和输出目录，避免长输出互相污染或覆盖；
- 只共享有界核心线索，不复制其他 Agent 的完整原始对话；
- 每个模型步骤前刷新共享线索板，因此运行时间较长的子 Agent 能读取同伴刚发布的证据；
- 停止任务后取消正在运行的模型/命令，排队任务不会继续启动；
- `parallel_tasks` 内任一任务带 `target_selector` 时，整体进入多目标风险确认流程。

推荐并行写法：

```json
{
  "action": "task",
  "parallel_tasks": [
    {
      "task_name": "log-analyst",
      "task_prompt": "只分析今天 auth.log：输出异常 IP、时间线和原始日志证据；完成后停止",
      "task_max_steps": 15
    },
    {
      "task_name": "network-analyst",
      "task_prompt": "只分析当前 established 连接和 DNS：输出可疑远端、PID 和证据；完成后停止",
      "task_max_steps": 12
    },
    {
      "task_name": "webshell-hunter",
      "task_prompt": "只检查 Web 根目录近期修改文件：输出路径、哈希和可疑代码证据；不得修改文件",
      "task_max_steps": 15
    }
  ]
}
```

`task_prompt` 应写清四件事：唯一范围、预期证据、禁止事项、停止条件。不要让多个子 Agent 都执行“全面扫描所有内容”，否则即使任务文本不完全相同，也会产生业务层面的重复劳动。

有依赖关系时使用两阶段流程：

1. 第一批并行完成日志、网络、文件取证；
2. 主 Agent 合并新线索、冲突与失败项并更新 TODO；
3. 第二批只针对第一批发现的 IP、路径或进程做定向复核。

#### 主 Agent 如何接收结果

并行任务结束后，主 Agent 会得到统一交接块：

- 总任务数、成功数、失败数、实际并发和去重数量；
- 每个子 Agent 的耗时、状态和限定长度的结果；
- 本轮新汇聚的核心线索及来源；
- 合并已验证事实、证据链、冲突、不确定项和失败任务的明确要求。

主 Agent 应先更新 TODO，再决定继续执行、发起第二阶段委派、询问用户或结束任务，不应把多个子 Agent 的原始长文本直接重复输出给用户。

### 5.4 外部 MCP

DeepSentry 支持 stdio 与 Streamable HTTP MCP Server。推荐使用结构化配置：

```yaml
mcp_server_configs:
  - name: fs
    type: stdio
    command: npx
    args: ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"]
    cwd: "/tmp"
    disabled: false

  - name: docs
    type: streamable_http
    url: https://mcp.example.com/mcp
    bearer_token_env_var: MCP_DOCS_TOKEN
    enabled_tools: [search, read]
    required: false
```

常用命令：

```text
/mcp status
/mcp import ~/Library/Application Support/Claude/claude_desktop_config.json
/mcp add fs npx -y,@modelcontextprotocol/server-filesystem,/tmp
/mcp login docs
/mcp resources docs
/mcp prompts docs
/mcp off fs
/mcp on fs
/mcp remove fs
```

远程 MCP 地址必须使用 HTTPS，localhost / 回环地址除外。Bearer Token 推荐只通过环境变量提供；OAuth 令牌只保留在当前进程，不写入配置文件。

## 10. 故障排查

| 现象 | 处理 |
|------|------|
| SSH 连接失败 | 检查端口（如 2222 非 22）、`-init` 重配密码 |
| LLM 429/超时 | 增大 `llm_retries`，检查 API 配额 |
| Agent 空转 | 减小任务粒度，或 `--resume` 继续 |
| 长会话频繁整理 | 检查是否持续产生超大命令输出；完整输出会卸载到 workspace，后续应按路径定向读取，不要反复整文件回灌 |
| 摘要失败提示 | Agent 会自动保留目标、上一版摘要、核心线索和最近消息继续；网络恢复后可直接继续，不必新建会话 |
| `/memory clues` 为空 | 当前尚未出现可提取的高信号标识或明确结论；这不表示任务历史丢失 |
| 子 Agent 重复劳动 | 把每个 `task_prompt` 改成互斥范围，并写清预期证据和停止条件；有依赖的任务拆成两批 |
| 并发任务看似变慢 | 多目标任务会主动降低并发，且内部目标展开限制为 1，这是为了避免 SSH/API 连接风暴 |
| 线索出现在错误目标 | 查看 `/memory clues` 的来源标签；来自其他 target 的线索必须在当前目标重新验证 |
| TUI/向导乱码或图标变方块 | Windows `cmd.exe` 和部分国产化终端会自动降级为 ASCII 图标但保留颜色；也可手动设置 `DEEPSENTRY_PLAIN=1` 或 `DEEPSENTRY_ASCII=1`。Windows Terminal/PowerShell 7 可用 `DEEPSENTRY_FANCY=1` 开启漂亮图标 |
| 需要纯无色输出 | 使用 `--no-color`，或设置 `NO_COLOR=1` / `DEEPSENTRY_NO_COLOR=1` |
| 向上翻阅时输入框仍覆盖日志，或回到底部后没有恢复 | 按 `Ctrl+End` / `G` 回到实时底部；确认启动的是当前构建产物。正常行为是离开底部后隐藏输入区，回到底部后恢复 |
| 中文输入法候选框或光标错位 | 先回到实时底部再输入，并确认启动的是当前构建产物；翻阅历史时真实输入应隐藏，候选框不会停留在日志区域 |
| Emoji 与文字重叠、表格或右边框错位 | 确认终端字体支持对应字符，并确认启动的是当前构建产物；不支持 Emoji 的终端可设置 `DEEPSENTRY_PLAIN=1` 降级 |
| 模型响应被错误显示为最终报告 | 需要补充信息或用户选择时应进入询问面板；若持续异常，检查模型兼容性、API 网关是否截断响应，并适当增大 `llm_timeout_sec` |
| `config_manage action=view` 提示未知动作 | 使用 `status` 查看安全配置摘要；`view/show/list/overview` 也作为兼容别名 |
| 出现 `root@host's password:` 并卡住 | 这是控制端裸 `ssh/scp/sftp` 的交互提示。按 `Ctrl+C` 中断，让 Agent 改用 `fleet_exec` / `fleet_file`；不要把目标密码输入 TUI 或拼进命令 |
| 本机 sudo 出现系统 `Password:` | DeepSentry 会暂时把终端交给 `sudo -v`。只在系统提示中输入；完成、取消或验证失败后会恢复 TUI。也可在空闲时执行 `/sudo` 预先验证 |
| 远程 sudo 报 `a password is required` | 出于凭据隔离，远程命令固定使用 `sudo -n`，不会复用 SSH 密码。请用 `visudo` 只为确有需要的命令配置最小范围 `NOPASSWD`，或改用具备权限的账号 |
| 编译后界面还是旧的 | 可能启动了旧二进制。推荐 `bash build.sh && cd build && ./deepsentry -c config.yaml`，或根目录 `go build -o deepsentry ./cmd/main.go ./cmd/usage.go ./cmd/survey_compat.go ./cmd/console_other.go && ./deepsentry -c config.yaml` |
| process_list 慢 | 已优化为限量读取；可调小 `limit` 参数 |
| FTP 批量命令失败 | FTP 不支持 shell，请使用 `fleet_file` 的文件动作 |
